# CLI 处理进度与转写重试可见性设计

日期：2026-06-23

## 1. 目标

为 CScribe 的长耗时处理流程增加清晰、稳定的终端进度反馈，并让 MiMo 转写重试对用户可见。

交互终端使用单行动态刷新，展示当前处理阶段、阶段耗时、转写完成数量以及最近发生的重试。非交互终端或输出重定向场景自动退化为普通日志，避免输出控制字符。

本次工作复用现有转写重试机制，不引入第二套重试流程。

## 2. 用户体验

同一时刻只保留当前进度行，阶段完成后由下一阶段替换，不在终端中累积完整阶段历史。

典型输出：

```text
⠹ 正在进行说话人分离｜已用时 01:42
⠸ 正在准备音频片段｜已完成 18 段
⠼ 正在转写音频片段 12/37｜片段 16 重试 2/3
⠴ 正在转写音频片段 13/37
✓ 正在写入结果
✓ 已完成｜36 成功｜1 失败｜耗时 03:26
```

展示规则：

- 音频探测、标准化、MPS 预检、说话人分离、片段准备、转写和结果写入均作为明确阶段。
- 无法获得可靠百分比的阶段使用旋转指示器和已用时间，不估算虚假的整体百分比。
- 转写阶段显示最终完成数量 `completed/total`。
- 重试期间附加显示最近发生的重试事件，例如 `片段 16 重试 2/3`。
- 片段只有在最终成功或最终失败后才计入 `completed`，重试过程不增加完成数量。
- 多个片段并发重试时，单行进度显示最近发生的重试事件。
- 警告、错误和 verbose 日志出现在进度行上方，动态行随后继续刷新。

## 3. 终端适配

使用 Rich 提供动态终端渲染，并新增其项目依赖。

当标准错误流是 TTY 时，启用单行动态进度。进度输出使用标准错误流，避免污染未来可能写入标准输出的机器可读内容。

当标准错误流不是 TTY 时：

- 不输出 spinner、回车覆盖或 ANSI 动画。
- 阶段开始、重试和最终结果使用普通日志。
- 高频的转写完成事件不逐条写日志，避免长录音产生大量噪声；可以按阶段开始和最终汇总记录。

Rich 的生命周期必须在 CLI 边界内可靠关闭。正常完成、配置错误、运行时异常和用户中断均不得留下未清理的动态行。

## 4. 架构

新增独立的进度报告边界，例如 `ProgressReporter`。它负责把业务事件转换为 TTY 动态显示或普通日志，不负责音频处理、重试判断或任务编排。

建议接口表达行为而非 Rich 组件细节：

- `start_stage(stage, detail=None)`
- `set_segment_total(total)`
- `segment_completed(success)`
- `segment_retrying(segment_index, retry_number, max_retries)`
- `finish(success_count, failure_count, elapsed_seconds)`
- `close()`

具体命名可在实现计划中按现有代码风格调整，但调用方不得依赖 Rich 的 task ID、console 或 renderable。

Reporter 作为可选依赖注入 pipeline 和 `MiMoTranscriber`：

- `run_pipeline` 通知阶段变化和最终汇总。
- 片段准备阶段可以通知已接受的片段数量。
- `MiMoTranscriber` 通知重试、最终成功和最终失败。
- 未提供 reporter 时使用无操作实现，现有调用方和测试不需要判断空值。

进度展示失败属于非关键故障。Reporter 应隔离渲染异常并退化为日志或无操作，不能中断转写或改变退出码。

## 5. 数据流

处理流程与进度事件对应如下：

```text
校验运行环境
  -> 探测音频
  -> 标准化音频
  -> 创建并运行可选 MPS 预检
  -> 说话人分离
  -> 清洗区间并准备音频片段
  -> 设置转写 total
  -> 并发转写
       -> 临时性失败：报告最近重试
       -> 最终成功或失败：completed + 1
  -> 提取关键词并写出结果
  -> 显示最终汇总
```

并发转写完成事件必须在事件接收端安全计数。动态显示只反映已经收到的最终结果数量，不根据请求启动数量或队列位置推算。

## 6. 转写重试规则

一次片段转写包含首次请求，以及首次失败后最多 3 次重试，总计最多 4 次请求。

仅以下临时性错误进入重试：

- 超时。
- 连接错误。
- HTTP 429。
- HTTP 5xx。

继续使用指数退避并保留现有轻微随机抖动，避免并发请求同时再次冲击服务。

认证错误、参数错误、其他明确的 HTTP 4xx、响应内容为空及本地编码错误不重试。

重试编号表示“第几次重试”，范围为 `1/3` 到 `3/3`，不把首次请求显示为一次重试。重试事件应在退避等待开始前发送，使用户立即知道任务仍在推进。

## 7. 最终失败与退出语义

某片段用尽重试后：

- 设置文本为 `[该片段识别失败]`。
- 保存简洁错误信息供调试输出使用。
- 将该片段计入完成数量和失败数量。
- 默认继续处理其他片段。
- 仍然生成正式 TXT。
- 整体退出码为 `2`。

`--fail-fast` 保留现有语义：片段用尽重试并最终失败后终止任务，不写正式 TXT。进度组件只展示这一结果，不改变失败策略。

致命的音频探测、标准化、说话人分离或输出写入错误仍返回退出码 `1`。

## 8. 日志协作

普通日志继续承载可追溯诊断，动态进度只提供当前状态。

- Rich 的日志输出能力或等价的安全挂接方式用于在动态行上方打印日志。
- 正常模式保持简洁。
- `--verbose` 可以展示设备决策、回退原因、重试异常类型和详细错误。
- 日志与进度均不得包含 MiMo API Key、Hugging Face Token、请求头或完整音频数据 URL。
- 最近重试提示可被后续重试或片段完成状态覆盖；可追溯信息仍保留在 verbose 日志中。

## 9. 测试

### 9.1 ProgressReporter 单元测试

- TTY 模式创建动态单行显示。
- 非 TTY 模式不生成 ANSI 动画或回车覆盖字符。
- 阶段切换更新阶段名称并重置适用的阶段状态。
- 说话人分离显示 spinner 和已用时间，不显示虚构百分比。
- 转写显示正确的 `completed/total`。
- 最近发生的并发重试覆盖旧的重试提示。
- 最终汇总包含成功数、失败数和总耗时。
- `close()` 可重复调用，并在异常路径清理动态行。
- 渲染故障不会传播到业务流程。

### 9.2 MiMoTranscriber 单元测试

- 首次请求成功时不发送重试事件。
- 超时、连接错误、429 和 5xx 最多重试 3 次。
- 重试事件依次报告 `1/3`、`2/3`、`3/3`。
- 认证错误、其他 4xx 和空响应不重试。
- 重试后成功只发送一次最终完成事件。
- 用尽重试后写入失败占位并发送一次最终失败事件。
- 并发完成顺序不影响最终结果按 segment index 排序。
- `--fail-fast` 只在片段最终失败后抛出。

### 9.3 Pipeline 与 CLI 测试

- 各处理阶段按实际执行顺序报告。
- 片段切分结束后才设置准确的转写总数。
- 部分失败仍写出 TXT、显示失败汇总并返回退出码 `2`。
- 致命异常关闭进度显示并保留完整错误消息。
- 非 TTY 执行产生可读普通日志。
- 未提供 reporter 的现有测试和业务调用继续工作。

## 10. 非目标

- 不提供基于历史速度的剩余时间预测。
- 不提供不可验证的整体百分比。
- 不记录每个片段的永久多行进度历史。
- 不增加断点续跑或失败片段的跨进程恢复。
- 不改变 MiMo 并发、限流或输出文件格式。
- 不改变 pyannote 的内部执行方式，也不尝试从其私有 API 推导百分比。

## 11. 验收标准

- 在交互终端运行时，用户始终能看到当前处理阶段。
- 说话人分离阶段持续显示活动状态和已用时间。
- 转写阶段准确显示最终完成片段数与总片段数。
- 临时性请求失败立即显示最近重试及其次数。
- 首次请求后最多执行 3 次重试。
- 最终失败片段保留占位，不导致其他片段缺失。
- 非交互输出无动画控制字符。
- 进度功能的任何故障都不影响音频处理结果或退出语义。
