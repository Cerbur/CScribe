# CLI 处理进度、流水线并发与断点恢复设计

日期：2026-06-23

## 1. 目标

为 CScribe 的长耗时处理流程增加清晰、稳定的终端进度反馈，并让 MiMo 转写重试对用户可见。切片和转录以有界流水线并发执行，中间产物持久化到确定性的 `/tmp` 工作目录，使中断后的相同任务能够自动续跑。

交互终端使用单行动态刷新，展示当前处理阶段、阶段耗时、转写完成数量以及最近发生的重试。非交互终端或输出重定向场景自动退化为普通日志，避免输出控制字符。

本次工作复用现有转写重试机制，不引入第二套重试流程。成功完成后主动清理工作目录，目标文件夹中只保留 TXT 和用户显式要求的调试 JSON。

## 2. 用户体验

同一时刻只保留当前进度行，阶段完成后由下一阶段替换，不在终端中累积完整阶段历史。

典型输出：

```text
⠹ 正在进行说话人分离｜已用时 01:42
⠸ 正在处理音频片段｜切片 18/37｜转写 12/37
⠼ 正在处理音频片段｜切片 20/37｜转写 12/37｜片段 16 重试 2/3
⠴ 正在处理音频片段｜切片 21/37｜转写 13/37
✓ 正在写入结果
✓ 已完成｜36 成功｜1 失败｜耗时 03:26
```

展示规则：

- 音频探测、标准化、MPS 预检、说话人分离、片段准备、转写和结果写入均作为明确阶段。
- 无法获得可靠百分比的阶段使用旋转指示器和已用时间，不估算虚假的整体百分比。
- 流水线阶段同时显示切片和最终转写完成数量。
- 重试期间附加显示最近发生的重试事件，例如 `片段 16 重试 2/3`。
- 片段只有在最终成功或最终失败后才计入转写完成数量，重试过程不增加完成数量。
- 多个片段并发重试时，单行进度显示最近发生的重试事件。
- 警告、错误和 verbose 日志出现在进度行上方，动态行随后继续刷新。

## 3. 终端适配

使用 Rich 提供动态终端渲染，并新增其项目依赖。

当标准错误流是 TTY 时，启用单行动态进度。进度输出使用标准错误流，避免污染未来可能写入标准输出的机器可读内容。

当标准错误流不是 TTY 时：

- 不输出 spinner、回车覆盖或 ANSI 动画。
- 阶段开始、重试和最终结果使用普通日志。
- 高频的转写完成事件不逐条写日志，避免长录音产生大量噪声；普通日志只记录阶段开始、重试和最终汇总。

Rich 的生命周期必须在 CLI 边界内可靠关闭。正常完成、配置错误、运行时异常和用户中断均不得留下未清理的动态行。

## 4. 架构

新增独立的进度报告边界 `ProgressReporter`。它负责把业务事件转换为 TTY 动态显示或普通日志，不负责音频处理、重试判断或任务编排。

接口表达行为而非 Rich 组件细节：

- `start_stage(stage, detail=None)`
- `set_segment_total(total)`
- `segment_sliced()`
- `segment_completed(success)`
- `segment_retrying(segment_index, retry_number, max_retries)`
- `finish(success_count, failure_count, elapsed_seconds)`
- `close()`

具体命名可在实现计划中按现有代码风格调整，但调用方不得依赖 Rich 的 task ID、console 或 renderable。

Reporter 作为可选依赖注入 pipeline 和 `MiMoTranscriber`：

- `run_pipeline` 通知阶段变化和最终汇总。
- 切片 worker 在切片完成并校验后通知完成数量。
- `MiMoTranscriber` 通知重试、最终成功和最终失败。
- 未提供 reporter 时使用无操作实现，现有调用方和测试不需要判断空值。

进度展示失败属于非关键故障。Reporter 应隔离渲染异常并退化为日志或无操作，不能中断转写或改变退出码。

## 5. 数据流

处理流程与进度事件对应如下：

```text
校验运行环境
  -> 探测音频
  -> 标准化音频
  -> 创建并运行可选 MPS 预检
  -> 说话人分离
  -> 清洗区间并设置稳定的 segment total
  -> 启动两个切片 worker
       -> 每个有效切片立即进入有界转录队列
  -> 多个转录 worker 并发处理
       -> 临时性失败：报告最近重试
       -> 最终成功或失败：转写 completed + 1
  -> 提取关键词并写出结果
  -> 显示最终汇总
```

切片和转写完成事件必须在事件接收端安全计数。动态显示只反映已经收到的有效切片和最终转写结果数量，不根据任务启动数量或队列位置推算。

## 6. 持久工作目录与任务身份

每个可恢复任务使用确定性的工作目录：

```text
/tmp/cscribe/<task-hash>/
├── manifest.json
├── task.lock
├── normalized.wav
├── preflight.wav
├── segments.json
├── audio/
│   ├── segment_0000.mp3
│   └── ...
└── transcripts/
    ├── segment_0000.json
    └── ...
```

`task-hash` 由以下信息的规范化表示计算：

- 输入文件的规范化绝对路径。
- 输入文件指纹。
- 目标 TXT 的规范化绝对路径。
- 影响标准化、说话人分离、切片或转写结果的关键参数。
- manifest schema 和处理算法版本。

输入文件指纹固定包含规范化绝对路径、文件大小、纳秒级修改时间，以及文件首尾各 1 MiB 内容的 SHA-256。小于 2 MiB 的文件对完整内容计算一次 SHA-256。

关键参数固定包括说话人数约束、语言、请求的 diarization 设备、关键词数量、模型 ID、音频编码常量和区间处理规则版本。并发数、限流、重试次数、verbose、fail-fast 和 debug-json 不改变缓存身份。

缓存位于 `/tmp`，因此恢复是尽力而为：操作系统可能随时清理缓存。缓存缺失时程序安全地从头运行，不视为异常。

目标索引位于 `/tmp/cscribe/targets/<sha256-normalized-output-path>.json`，内容为当前 `task-hash`。若输入内容或关键参数变化产生新 hash，启动时根据索引清理该目标对应的旧工作目录，原子更新索引，再创建新任务。

## 7. Manifest 与恢复规则

`manifest.json` 是恢复状态的权威来源，至少记录：

- schema 和处理算法版本。
- 输入指纹、目标路径和关键参数快照。
- 标准化音频状态。
- 说话人分离状态、设备决策和稳定片段清单。
- 每段切片路径、状态、字节数及片段身份。
- 每段转录状态、成功文本或最后失败的简洁原因。
- 最终输出状态。

状态文件使用临时文件加原子替换写入。切片和片段转录结果也必须先写临时文件，再原子替换为正式缓存文件。

恢复时遵循以下规则：

- manifest、输入指纹和关键参数完全匹配时自动续跑，不要求 `--resume`。
- 标准化音频有效时复用，不再次运行 FFmpeg 标准化。
- 说话人分离结果有效时复用，不再次加载或执行 pyannote。
- 有效切片复用；缺失、空文件、身份不匹配或损坏的切片单独重建。
- 已成功转录的片段永不重复请求。
- pending 和 failed 片段进入本次转录队列。
- 每次新进程启动时，待转录片段重新获得首次请求加最多 3 次重试；重试次数不跨进程累计。
- 恢复后的最终结果仍按开始时间、结束时间和不可变 `segment_id` 排序。

仅凭文件存在不能认定阶段完成。程序必须同时验证 manifest 状态和对应产物的最低完整性。

## 8. 任务锁与并发进程

每个任务使用进程锁，避免两个进程共享同一缓存并重复发送 API 请求。

- 第一个进程原子获取锁。
- 相同任务再次启动时，若持锁进程仍活跃，后启动者立即返回清晰错误。
- 锁记录 PID、进程启动时间和随机运行 ID。仅当 PID 与启动时间仍匹配时视为活跃；否则视为陈旧锁并安全接管任务。
- 正常完成、异常退出和 `Ctrl+C` 均尽力释放锁。
- 锁释放失败不得删除可恢复产物。

锁文件本身不能只依赖“存在即活跃”，必须包含足以检查本机进程存活的信息。

## 9. 流水线并发

说话人分离完成并生成稳定片段清单后，启动生产者—消费者流水线：

- 固定启动最多两个 FFmpeg 切片 worker。
- 切片完成并通过大小校验后，立即进入转录队列。
- 转录 worker 数量由现有 `--concurrency` 控制。
- 转录队列容量固定为 `max(2, 2 × --concurrency)`，形成背压，避免长录音一次性堆积大量待发送数据。
- 从缓存恢复的有效切片可以直接进入转录队列，不再占用切片 worker。
- 已成功转录的片段不进入任何工作队列，只计入恢复后的初始完成数量。

每个片段使用不可变 `segment_id` 标识，普通片段形如 `s0007`。若切片超过 Base64 限制，父片段在发布到转录队列前替换为连续子片段，子 ID 形如 `s0007.0`、`s0007.1`。该 manifest 更新必须串行且原子完成；已经发布的其他片段身份不变。展示总数随子片段清单扩展而增加，最终输出按开始时间、结束时间和 `segment_id` 排序，而不是依赖可漂移的数组位置。

切片失败仅影响对应片段，本次运行不自动重试 FFmpeg。该片段写入 `[该片段识别失败]`、将切片状态记为 failed 并让其他片段继续；下次续跑时 failed 切片重新进入切片队列。标准化或说话人分离失败仍是关键阶段失败。

## 10. 转写重试规则

一次片段转写包含首次请求，以及首次失败后最多 3 次重试，总计最多 4 次请求。

仅以下临时性错误进入重试：

- 超时。
- 连接错误。
- HTTP 429。
- HTTP 5xx。

继续使用指数退避并保留现有轻微随机抖动，避免并发请求同时再次冲击服务。

认证错误、参数错误、其他明确的 HTTP 4xx、响应内容为空及本地编码错误不重试。

重试编号表示“第几次重试”，范围为 `1/3` 到 `3/3`，不把首次请求显示为一次重试。重试事件应在退避等待开始前发送，使用户立即知道任务仍在推进。

## 11. 最终失败、缓存清理与退出语义

某片段用尽重试后：

- 设置文本为 `[该片段识别失败]`。
- 保存简洁错误信息供调试输出使用。
- 将该片段计入完成数量和失败数量。
- 默认继续处理其他片段。
- 仍然生成正式 TXT。
- 整体退出码为 `2`。
- 保留 `/tmp` 工作目录，使下次相同命令只重试失败片段。

`--fail-fast` 保留现有语义：片段用尽重试并最终失败后终止任务，不写正式 TXT。进度组件只展示这一结果，不改变失败策略。

致命的音频探测、标准化、说话人分离或输出写入错误仍返回退出码 `1`。

全部片段成功且最终输出原子写入后：

- 删除对应 `/tmp/cscribe/<task-hash>/` 工作目录。
- 清理该目标路径的任务索引。
- 目标目录中只留下正式 TXT。
- 用户显式指定 `--debug-json` 时，额外保留 `.segments.json`。

部分失败时仍原子生成包含 `[该片段识别失败]` 的 TXT，但保留工作目录。后续续跑若全部成功，则原子替换原 TXT 并清理缓存。

`Ctrl+C` 时停止接收新切片和转录任务，取消尚未开始的工作，等待或取消正在运行的任务，原子保存已完成状态，释放任务锁并保留工作目录。实现必须避免把正在写入的缓存误标为完成。

## 12. 日志协作

普通日志继续承载可追溯诊断，动态进度只提供当前状态。

- Rich 的日志输出能力或等价的安全挂接方式用于在动态行上方打印日志。
- 正常模式保持简洁。
- `--verbose` 展示设备决策、回退原因、重试异常类型和详细错误。
- 日志与进度均不得包含 MiMo API Key、Hugging Face Token、请求头或完整音频数据 URL。
- 最近重试提示可被后续重试或片段完成状态覆盖；可追溯信息仍保留在 verbose 日志中。

## 13. 测试

### 13.1 ProgressReporter 单元测试

- TTY 模式创建动态单行显示。
- 非 TTY 模式不生成 ANSI 动画或回车覆盖字符。
- 阶段切换更新阶段名称并重置适用的阶段状态。
- 说话人分离显示 spinner 和已用时间，不显示虚构百分比。
- 转写显示正确的 `completed/total`。
- 流水线阶段同时显示正确的切片与转写计数。
- 最近发生的并发重试覆盖旧的重试提示。
- 最终汇总包含成功数、失败数和总耗时。
- `close()` 可重复调用，并在异常路径清理动态行。
- 渲染故障不会传播到业务流程。

### 13.2 MiMoTranscriber 单元测试

- 首次请求成功时不发送重试事件。
- 超时、连接错误、429 和 5xx 最多重试 3 次。
- 重试事件依次报告 `1/3`、`2/3`、`3/3`。
- 认证错误、其他 4xx 和空响应不重试。
- 重试后成功只发送一次最终完成事件。
- 用尽重试后写入失败占位并发送一次最终失败事件。
- 并发完成顺序不影响最终结果按时间和 `segment_id` 排序。
- `--fail-fast` 只在片段最终失败后抛出。

### 13.3 Pipeline 与 CLI 测试

- 各处理阶段按实际执行顺序报告。
- 说话人区间稳定后设置准确的片段总数。
- 切片最多同时运行两个。
- 有效切片完成后立即进入转录，不等待全部切片结束。
- 转录并发仍受 `--concurrency` 限制。
- 有界队列在转录较慢时产生背压。
- 超限切片使用稳定子 ID 扩展 manifest，其他片段身份不发生变化。
- FFmpeg 切片失败不阻止其他片段，下次续跑会重新切片。
- 部分失败仍写出 TXT、显示失败汇总并返回退出码 `2`。
- 致命异常关闭进度显示并保留完整错误消息。
- 非 TTY 执行产生可读普通日志。
- 未提供 reporter 的现有测试和业务调用继续工作。

### 13.4 恢复与缓存测试

- 中断后重新执行相同命令会自动加载同一任务。
- 有效的标准化音频和说话人分离结果不会重复生成。
- 有效切片不会重复运行 FFmpeg。
- 成功片段不会重复调用 MiMo。
- failed 和 pending 片段在新进程中重新获得完整请求次数。
- 缺失、空文件、半写或身份不匹配的切片会单独重建。
- manifest 半写不会覆盖最后一个有效版本。
- 输入内容或关键参数变化时清理旧目标缓存并从头运行。
- 活跃任务锁阻止第二个进程；陈旧锁可以安全接管。
- `Ctrl+C` 后已完成结果可恢复，正在进行的结果不会误标成功。
- 部分失败保留缓存，全部成功删除缓存和目标索引。
- `/tmp` 缓存被系统清理后程序可安全从头运行。

## 14. 非目标

- 不提供基于历史速度的剩余时间预测。
- 不提供不可验证的整体百分比。
- 不记录每个片段的永久多行进度历史。
- 不改变 MiMo 并发、限流或输出文件格式。
- 不改变 pyannote 的内部执行方式，也不尝试从其私有 API 推导百分比。
- 不保证 `/tmp` 中的恢复状态跨重启或系统清理长期存在。
- 不支持跨机器恢复或把工作目录作为可移植任务包。
- 不允许多个进程协同处理同一个任务。

## 15. 验收标准

- 在交互终端运行时，用户始终能看到当前处理阶段。
- 说话人分离阶段持续显示活动状态和已用时间。
- 转写阶段准确显示最终完成片段数与总片段数。
- 临时性请求失败立即显示最近重试及其次数。
- 首次请求后最多执行 3 次重试。
- 最终失败片段保留占位，不导致其他片段缺失。
- 最多两个 FFmpeg 切片与 MiMo 转录形成有界重叠流水线。
- 相同任务中断后自动续跑，并复用所有已验证的前序产物和成功文本。
- 相同任务的第二个活跃进程被拒绝。
- 部分失败保留工作目录，全部成功后主动清理。
- 正常成功时目标目录只保留 TXT 和用户显式要求的调试 JSON。
- 非交互输出无动画控制字符。
- 进度功能的任何故障都不影响音频处理结果或退出语义。
